# 前端预览检查

这份材料只回答一个问题：

- 主代理如何避免被旧进程、旧产物、旧端口误导

它适用于所有需要本地预览的前端或用户可见交付，不绑定具体框架。

## 1. 默认原则

前端验证不要直接相信：

- 子代理口头结论
- 旧浏览器标签页
- 不带端口检查的本地预览

主代理至少要拿到一条可回链的运行证据。

## 2. 启动服务前的固定顺序

按这个顺序检查：

1. 先查目标端口是否已有监听
2. 如果已有监听，先判断是否可复用
3. 如果不可确定是否为最新产物，换新端口启动
4. 先完成 build，再启动生产预览
5. 再做页面探测

重点：

- 不要默认复用旧端口
- 不要默认相信“已经有服务在跑”

## 3. 推荐的最小验证链

### A. 构建验证

至少确认：

- build 通过
- 没有新增的类型错误或构建错误

### B. 运行验证

至少确认：

- 预览进程已启动
- 关键路由返回 200
- 页面输出包含本轮最关键的结构信号

### C. 结构验证

优先检查：

- 标题
- 关键动作文案
- 关键类名或结构块
- 关键折叠区或模式切换入口

## 4. 页面探测优先级

优先顺序固定为：

1. 主入口页
2. 本轮改动最重的主页面
3. 一条关键详情页、工作页或阅读页
4. 必要时再测补充页

不要一开始就探测所有页面。

## 5. 何时必须换新端口

满足任一条件就换新端口：

- 不确定当前监听进程来自哪次构建
- 本轮改动包含全局样式或路由结构
- 子代理报告和当前页面表现不一致
- 旧实例可能缓存了过时产物

## 6. 何时主代理必须亲自验证

满足任一条件时，主代理不能只看 verifier 结论：

- 本轮任务有明显主观性，例如 UI、引导、排版
- 用户前一轮已经指出“看起来不对”
- 子代理只汇报“感觉更好了”，没有结构证据
- verifier 只验证了构建，没有验证关键页面输出

## 7. 结构问题与视觉问题的证据优先级

### 结构问题

优先：

1. 文案
2. 路由
3. DOM / HTML
4. 类名 / 结构块

### 视觉问题

优先：

1. 局部页面探测
2. 局部截图
3. 必要时再看整页截图

不要默认整页截图。

## 8. 验收记录最少要留下什么

主代理至少记录：

- 使用了哪个端口
- 探测了哪些路由
- 哪些关键信号已经出现
- 构建是否通过
- 是否亲自复核

## 9. 常见失败模式

### 1. 看的是旧服务

表现：

- 页面和代码明显不一致

纠正：

- 先查端口，再换新端口起服务

### 2. 只验证 build，没验证页面

表现：

- 代码能编过，但用户看不到结果

纠正：

- 必补关键路由探测

### 3. 只看截图，不看结构

表现：

- 花很多上下文讨论视觉，但问题难定位

纠正：

- 先用结构证据缩小范围，再用截图确认视觉问题
