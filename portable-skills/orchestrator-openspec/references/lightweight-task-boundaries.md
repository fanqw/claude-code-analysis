# 轻量任务边界案例

这份文档回答两个问题：

- 哪些任务可以不进入 OpenSpec
- 哪些任务看起来小，但实际上不应被误判为轻量任务

## 可视为轻量任务的典型情况

### 1. 单文件局部修正

- 写入范围清楚
- 风险低
- 无需方案裁定
- 单轮可收口

### 2. 纯文档修订

- 不改变稳定行为约束
- 不涉及运行结果
- 不涉及多角色并行

### 3. 小范围样式或文案修正

- 目标清楚
- 影响面可控
- 不需要多轮规格冻结

### 4. 明确的低风险脚本修补

- 原因明确
- 修法明确
- 主代理可直接静态判断风险边界

## 不应误判为轻量任务的情况

### 1. 看似改动小，但需要方案裁定

只要需要先决定“应该怎么做”，而不是直接改，就应考虑进入 OpenSpec。

### 2. 看似只改几个文件，但跨多个子系统

文件数少，不代表复杂度低。

### 3. 看似只是界面改动，但会改变关键路径或长期规则

这类改动可能影响行为契约或后续判断，不应简单按轻量处理。

### 4. 看似一次能改完，但预期要多轮返工或多轮验收

这类任务仍应进入 OpenSpec。

## 快速判断法

满足全部条件时，通常可以不进 OpenSpec：

- 不需要方案裁定
- 不跨多个子系统
- 不需要多角色并行
- 单轮可收口
- 不改变长期稳定约束

若任一条件不满足，主代理至少应重新判断是否进入 OpenSpec。
