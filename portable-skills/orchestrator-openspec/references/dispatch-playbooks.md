# 主代理派单剧本

这份文档回答两个问题：

- 主代理在什么情况下应派哪类子代理
- 主代理在什么情况下必须追加验证或亲自复核

它服务于派单决策，不替代 `subagent-roles.md`、`model-selection.md` 或具体 prompt 模板。

## 一、默认派单顺序

主代理默认按下面的顺序判断：

1. 先判断当前问题是否已经足够清楚
2. 若不清楚，先派 `explorer`
3. 方案冻结后，再派 `builder`
4. 有运行风险或交付门槛时，再派 `verifier`
5. 若结果存在冲突或高风险，主代理亲自复核

不要一开始就把所有角色全部派出去。主代理应先确定当前最小必要组合。

## 二、标准派单组合

### 1. 只读澄清组合

适用条件：

- 需求不够清楚
- 代码边界不清楚
- 存在多个可选方案
- 主代理还不能冻结方案

推荐组合：

- `explorer`

主代理动作：

- 要求 explorer 输出边界、风险、候选方案
- 根据输出更新方案裁定
- 在方案未冻结前，不派 builder

### 2. 先探后建组合

适用条件：

- 任务中等复杂
- 需要先读代码或读规格再实施
- 写入范围已大致可控，但仍需先确认

推荐组合：

- `explorer`
- `builder`

派发顺序：

1. 先派 explorer
2. 主代理冻结方案
3. 再派 builder

禁止做法：

- explorer 结果未审完就让 builder 开始写

### 3. 实现加技术验收组合

适用条件：

- 有明确可执行方案
- 会产生运行结果
- 需要确保 build、启动、测试或接口行为正常

推荐组合：

- `builder`
- `verifier`

派发顺序：

1. builder 实现
2. 主代理初审
3. verifier 独立验证
4. 主代理综合裁定

禁止做法：

- 让 builder 的自测代替 verifier

### 5. 完整交付组合

适用条件：

- 中大型变更
- 同时涉及实现和运行稳定性
- 预计至少一轮独立验证

推荐组合：

- `explorer`
- `builder`
- `verifier`

推荐顺序：

1. explorer 澄清边界
2. 主代理冻结方案与任务包
3. builder 实现
4. verifier 做技术验收
5. 主代理综合结果并最终裁定

## 三、何时必须加 verifier

满足任一条件时，默认必须追加 `verifier`：

- 改动会影响构建、启动、部署或运行链
- 改动涉及命令、接口、状态变更或数据流
- 改动结果不能只靠静态 diff 判断
- 交付要求里明确写了验证命令、验证路径或预期输出
- 子代理自测和主代理预期存在差异

## 四、何时主代理必须亲自复核

满足任一条件时，主代理不能只依赖子代理结论：

- 两个子代理反馈互相冲突
- builder 与 verifier 结论不一致
- 当前改动有明显用户可见风险，但现有证据仍不足
- 当前任务处于关键路径
- 需要做最终方案裁定
- 主代理对结果可信度不足

主代理亲自复核的方式可包括：

- 亲自阅读关键 diff
- 亲自运行关键命令
- 亲自检查关键路径
- 亲自确认规格与实现是否对齐

## 五、何时可以跳过某类角色

### 可以不派 explorer

满足全部条件时可跳过：

- 目标清楚
- 写入范围清楚
- 方案基本无歧义
- 风险较低

### 可以不派 verifier

满足全部条件时可跳过：

- 没有运行面变化
- 只做只读产物或纯文档修改
- 主代理能直接静态确认结果

## 六、驳回与返工规则

主代理收到子代理结果后，只能给三种裁定：

- `accepted`
- `revise`
- `blocked`

其中：

- `accepted`：默认关闭；只有明确存在紧接的下一项任务时才短暂保留
- `revise`：必须带具体驳回原因和修改方向
- `blocked`：说明当前依赖、前置条件或规格仍未满足

不要使用模糊反馈，例如“再看看”“感觉不太对”。

## 七、推荐的最小派单字段

主代理派单时，建议至少包含这些字段：

- 任务目标
- 所属 change 或任务包
- 当前阶段
- 角色
- 模型选择
- 写入范围
- 禁止修改范围
- 前置依赖
- 产出物
- 验收条件
- 是否需要独立验证

## 八、子代理可见性与关闭

主代理在派发子代理时，默认必须公开：

- 角色
- 可识别名称
- 当前任务范围

主代理在任务结束时，默认必须决定：

- 是否继续复用
- 是否立即关闭

如果没有明确的下一项紧接任务，就关闭，不要把空闲子代理留在会话里。

## 九、推荐的主代理工作节奏

每一轮推荐采用这个固定节奏：

1. 主代理锁定本轮唯一目标
2. 判断本轮最小必要角色组合
3. 派发任务
4. 审核结果
5. 追加验证或走查
6. 必要时亲自复核
7. 产出轮次总结

重点不是“派了多少子代理”，而是“每一轮是否清楚、可控、可验收”。

## 十、协议化扩展原则

主代理新增角色、模板或脚本时，优先在既有协议内扩展，而不是临时发明协作方式。

至少保持这些协议稳定：

- 角色协议
- 派单字段协议
- 验收裁定协议
- 生命周期协议
- 模型选择协议

如果当前需求无法落在既有协议内，主代理应先补规则，再派单，而不是让某个子代理临场定义新做法。
