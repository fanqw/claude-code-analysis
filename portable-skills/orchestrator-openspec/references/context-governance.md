# 长任务上下文治理

这份文档回答一个问题：

- 多轮、多阶段、多子代理的长任务，主代理如何不失焦地继续推进

它不替代阶段门禁，也不替代验收规则。  
它只负责定义“上下文该保留什么、压缩什么、如何续接”。

## 一、治理目标

长任务上下文治理的目标不是保留所有细节，而是保留继续推进所需的最小必要上下文。

主代理每轮都应确保后续还能快速回答：

- 这一轮的唯一目标是什么
- 当前处于哪个阶段
- 当前 change 走到哪里
- 当前任务包从哪里来
- 当前最关键阻塞项是什么
- 下一步应该做什么

## 二、最小持续上下文

每轮结束后，至少保留这六类上下文：

1. 当前轮唯一目标
2. 当前阶段位置
3. 当前 change / 规格入口
4. 当前任务包或待办来源
5. 当前阻塞项与风险
6. 下一步推荐动作

如果缺少其中任一项，下一轮通常会重新做背景恢复，导致效率下降。

## 三、应该保留什么

主代理默认应保留：

- 当前 change 名称或入口
- 当前阶段
- 当前已接受的任务结果
- 当前仍未关闭的阻塞项
- 已冻结但后续仍会引用的决定
- 直接影响下一轮判断的验证结果

## 四、应该压缩什么

主代理默认应压缩，而不是原样保留：

- 已经关闭的问题讨论
- 被否决的备选方案细节
- 已完成且不会再引用的中间操作过程
- 重复出现的背景说明
- 已经被规格吸收的临时口头约定

原则：

- 保留结论，不保留无关过程
- 保留约束，不保留冗余描述
- 保留下一步所需信息，不保留已经失效的噪声

## 五、轮次切换时必须更新的内容

每次从一轮进入下一轮，主代理至少更新：

- 当前轮唯一目标是否完成
- 当前阶段是否变化
- 当前 change 状态是否变化
- 当前阻塞项是否解除
- 下一步是否变化
- 是否需要新增或关闭子代理

如果这些内容没有被更新，下一轮容易沿着过期上下文继续执行。

## 六、上下文压缩时的固定顺序

主代理建议按这个顺序压缩：

1. 先确认本轮目标
2. 再确认本轮结论
3. 再确认仍然有效的约束
4. 再确认仍未关闭的阻塞项
5. 最后明确下一步

不要反过来先写“做了很多什么”，再试图从中找下一步。

## 七、上下文失效判定

满足任一情况时，当前上下文应被视为失效或部分失效：

- 当前轮唯一目标已经无法描述
- 当前阶段与实际工作内容不一致
- 当前 change / 任务包入口已过期或无法回链
- 子代理在继续解决已被否决或已关闭的问题
- 下一步仍然依赖已经失效的约束
- 主代理无法解释为什么还在当前阶段继续推进

## 八、失效后的强制回退规则

一旦判定上下文失效，主代理不应继续沿原路径推进。

默认动作：

1. 先停止继续加派任务
2. 重新写当前轮唯一目标
3. 重新确认当前阶段
4. 重新确认当前有效的 change / 任务入口
5. 必要时回退到最近可修复问题的阶段

如果失效发生在：

- 规格前：回退到 `Intent Lock` 或 `Solution Exploration`
- 派单前：回退到 `Decision & Brief Freeze`
- 验收前：回退到 `Parallel Execution` 或 `Acceptance & Verification`

## 九、长任务中的常见失焦信号

满足任一情况时，说明上下文治理已经开始失焦：

- 同一轮里出现多个并列目标
- 当前阶段说不清
- 当前 change 或任务包入口说不清
- 子代理在解决已经过期的问题
- 主代理反复重新解释同一段背景
- 下一步仍然是“再看看”“继续推进”这类模糊表达

出现这些信号时，应先做上下文收口，再继续执行。

## 十、推荐的统一上下文包

主代理在轮次收口和下一轮续接时，建议统一输出：

- 当前轮唯一目标
- 当前阶段
- 当前 change / 规格入口
- 当前任务包来源
- 当前完成情况
- 当前阻塞项
- 当前保留约束
- 当前验证目标
- 下一步

如果需要标准骨架与脚手架，只使用这一组：

- [context-packet-template.md](context-packet-template.md)
- `../scripts/generate-context-packet.sh`

## 十一、与其他能力的关系

上下文治理不是独立能力，它支撑：

- 统一执行内核
- 规格先于执行
- 权限与裁定门禁
- 证据驱动验收与可追溯性

如果上下文治理失效，其他能力即使存在，也会在多轮任务中变弱。
