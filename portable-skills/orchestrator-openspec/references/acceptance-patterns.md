# 主代理验收模式

这份文档回答三个问题：

- 什么情况下应给出 `accepted`
- 什么情况下应给出 `revise`
- 什么情况下应给出 `blocked`

它服务于主代理验收裁定，不替代 `dispatch-playbooks.md` 或 `opsx-audit-guide.md`。

## 一、验收总原则

主代理验收时，不只看“有没有产出”，而是至少同时判断：

- 是否符合当前任务目标
- 是否符合当前规格或任务包
- 是否满足验收条件
- 是否存在缺失的独立验证
- 是否有足够证据支撑裁定

这套 skill 的验收属于“证据驱动验收”，不是口头共识验收。

不要把“已经提交结果”误当成“已经可接受”。

## 二、三种标准裁定

主代理只使用三种标准裁定：

- `accepted`
- `revise`
- `blocked`

不要使用含义模糊的表达替代正式裁定。

## 三、何时使用 accepted

满足全部条件时，主代理可给出 `accepted`：

- 结果覆盖了当前任务包要求
- 没有明显偏离规格
- 需要的独立验证已经完成，或可明确说明不需要
- 没有关键阻塞项
- 当前证据足以支撑接受决定

推荐写法：

- 已完成什么
- 依据什么接受
- 是否还有后续任务

示例骨架：

- 裁定：`accepted`
- 结论：结果已覆盖当前任务目标
- 依据：已满足任务包要求，且独立验证已通过
- 后续：默认关闭该子代理；只有明确存在紧接的下一项任务时才保留

## 四、何时使用 revise

满足任一条件时，主代理优先给出 `revise`：

- 结果部分完成，但未完全达标
- 结果方向基本正确，但细节不满足要求
- 存在遗漏项、越界项或格式问题
- 缺少必须的验证或证据
- 可通过继续修改解决

`revise` 必须包含：

- 问题点
- 为什么不通过
- 修改方向
- 复提时需要带回的证据

不合格写法：

- “再改一下”
- “感觉还不太对”

合格骨架：

- 裁定：`revise`
- 问题：当前结果缺少哪些要求
- 原因：为什么这些问题阻止通过
- 修改方向：下一步需要补什么
- 复提要求：提交时需要带回哪些证据

## 五、何时使用 blocked

满足任一条件时，主代理优先给出 `blocked`：

- 当前依赖未满足
- 规格未冻结
- 前置条件未完成
- 缺少必要资源、环境或输入
- 继续修改也无法在当前条件下推进

`blocked` 的重点不是指出实现问题，而是指出：

- 卡在哪里
- 卡点属于哪一类依赖
- 由谁先解除阻塞

合格骨架：

- 裁定：`blocked`
- 阻塞项：当前缺失什么前置条件
- 影响：为什么现在不能继续
- 下一步：谁需要先解决什么

## 六、验收证据模式

主代理验收时，优先使用低成本、可复查的证据：

1. 任务包、规格、diff、路径、文案、DOM
2. 命令结果、测试结果、接口结果、预览结果
3. 局部截图
4. 整页截图

如果当前裁定依赖截图，优先使用局部截图，不要默认依赖整页截图。

主代理最终应能把裁定回链到：

- 当前 change 或规格入口
- 当前任务包
- 当前验证结果
- 当前轮次总结

## 七、独立验证要求

满足以下情况时，不要只看 builder 结果就直接接受：

- 结果会影响运行行为
- 结果有用户可见变化
- 结果不能只靠静态阅读判断
- 主代理对结果可信度不足

此时应追加：

- `verifier`
- 或主代理亲自复核

## 八、主代理常见误判

主代理应避免这些误判：

- 因为时间紧，就跳过必要验证
- 因为结果看起来接近，就提前接受
- 因为子代理很自信，就忽略证据不足
- 因为发现问题不大，就不给明确驳回方向

关键原则：

- 验收不是情绪判断，而是基于任务包和证据的裁定

## 九、推荐的验收输出结构

主代理每次验收，建议固定输出这四部分：

1. 裁定
2. 结论
3. 依据
4. 下一步

推荐骨架：

- 裁定：`accepted | revise | blocked`
- 结论：一句话说明当前状态
- 依据：对应哪些要求、哪些验证、哪些证据
- 下一步：后续动作是什么

## 十、何时主代理必须亲自下场

满足任一条件时，主代理应亲自复核：

- 两个子代理结论冲突
- 规格和实现是否对齐存在争议
- 当前处于关键路径
- 当前裁定会直接决定是否收口
- 主代理无法仅靠现有证据做出高可信决定

主代理亲自复核后，仍然应输出正式裁定，而不是口头模糊结论。
