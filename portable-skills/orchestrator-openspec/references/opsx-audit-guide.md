# OpenSpec 审查与派单说明

这个文件不是 OpenSpec 教程，而是告诉主代理如何把 OpenSpec 原生工件纳入编排流程。

## 核心原则

- OpenSpec 是规格与证据底座
- 主代理是流程控制器
- 子代理不绕过规格直接自由发挥

## 主代理在 OpenSpec 中关心什么

主代理最关心的不是工件长什么样，而是它们是否已经足够支持：

- 目标锁定
- 方案冻结
- 子代理派单
- 验收回链

## 派单前至少确认

主代理在进入实施前，至少确认：

- 当前 change 已存在
- 当前方案已冻结到可以拆任务
- 当前任务工件里能定位每个任务的：
  - 目标
  - 负责人角色
  - 写入范围
  - 验收条件

如果做不到，先继续完善 OpenSpec 工件，而不是直接开工。

## Builder 派单来源

Builder 的任务应从当前 change 的任务工件提炼。

主代理下发时要再补三类局部信息：

- 当前轮次目标
- 禁止修改范围
- 本轮依赖和先后顺序

## Verifier 派单来源

Verifier 的任务来源于：

- 当前 change 的验收要求
- 主代理补充的运行验证目标

常见验证面：

- build
- start
- curl
- route
- 用户可见表面
- 测试命令

## 主代理验收时要回查什么

主代理最后至少回查：

- 实现是否仍符合初始目标
- 实现是否偏离当前规格
- 任务是否按拆分完成
- 验证结果是否支持接受

## 何时要求补 spec delta

只有这些情况才要求补 spec delta：

- 对外接口变化
- 行为契约变化
- 长期规则变化
- 会被后续多轮持续引用的稳定规范变化

局部重构、展现层结构调整、一次性体验优化，一般不要求 spec delta。
