# 学习指引网站 3 PM 子代理协作规范

## 1. 目标

本规范用于约束学习指引网站的需求讨论过程，避免多代理协作退化成自由脑暴。

协作目标只有一个：

- 把当前网站收敛成一条清晰的学习主线
- 把页面职责收敛为单一主职责
- 把产品决策沉淀为可直接交给设计和开发执行的统一方案

本规范不用于直接改代码，不用于视觉试验，也不用于增长策略讨论。

## 2. 适用范围

本流程只用于产品方案阶段，聚焦以下页面和相邻结构：

- `/`
- `/map`
- `/learn/[slug]`
- `/architecture`
- `/analysis`
- `/sources`

默认讨论目标用户：

- 第一次进入该学习站的新手或半熟悉用户
- 需要先建立 Claude Code 主线认知，再进入源码和 analysis 证据层的学习者

## 3. 当前问题定义

结合当前站点实现，以下问题视为默认已知问题：

- 多个页面同时承担导览、解释、证据、跳转职责
- 首页和地图页的信息过多，首屏入口不够单一
- 章节页承载内容过满，容易把导学、细节和证据混在一起
- `/analysis` 与 `/sources` 的证据中心定位不够明确
- 当前体验更像“信息总览”，而不是“有节奏的学习引导”

## 4. 角色与边界

### 4.1 UI 交互 PM

负责：

- 页面层级
- 首屏结构
- 导航方式
- 阅读节奏
- 信息密度
- 状态反馈
- 响应式阅读

不负责：

- 内容事实
- 章节顺序
- 产品范围
- 技术实现细节

禁止输出：

- 高级
- 现代
- 精致
- 沉浸
- 有氛围

所有建议必须映射到页面结构、导航、状态反馈、组件层级或移动端阅读体验。

### 4.2 内容引导 PM

负责：

- 学习主线
- 信息顺序
- 首屏讲什么
- 哪些内容必须延后
- 哪些证据不该抢首屏
- 术语解释规则
- 信息负担控制

不负责：

- 视觉风格
- 组件造型
- 技术方案

硬约束：

- 每页只能定义一个主职责
- 优先减少认知负担，不追求信息完整

### 4.3 整体规划 PM

负责：

- 问题定义
- 页面职责划分
- 优先级
- 轮次节奏
- 分歧裁决
- 停机条件
- 最终统一文档

不负责：

- 逐段文案优化
- 具体组件样式细节

硬约束：

- 必须明确不做项
- 第 3 轮后必须冻结
- 出现冲突时必须裁决，不能并列保留多个方向

## 5. 协作接口

所有子代理必须共享同一套输入和输出接口。

### 5.1 Shared Brief

字段：

- `target_user`
- `current_goal`
- `page_scope`
- `current_pain_points`
- `current_page_roles`
- `must_keep`
- `can_delay`
- `constraints`
- `last_round_consensus`
- `round_question`

作用：

- 统一输入上下文
- 禁止代理凭空扩展讨论边界

### 5.2 Round Prompt

规则：

- 每轮只允许一个主题
- 每轮只允许解决一个问题

合法主题示例：

- 首页首屏是否只保留唯一推荐入口
- `/map` 是否只保留阶段推进职责
- `/learn/[slug]` 是否把证据区后置
- `/analysis` 是否彻底降级为证据中心

### 5.3 Decision Ledger

字段：

- `question`
- `consensus`
- `conflicts`
- `decision`
- `non_goals`
- `acceptance_criteria`
- `next_round_question`

作用：

- 记录每轮裁决
- 防止下一轮重开已关闭问题

### 5.4 Final Spec

要求：

- 只允许一份主文档
- 附录保留轮次记录、分歧和裁决
- 所有结论都必须能映射到页面职责、内容顺序或交互规则

## 6. 固定迭代机制

### 6.1 轮次结构

每轮固定三步：

1. 整体规划 PM 发布本轮主题、范围、约束和成功标准
2. UI PM 与内容引导 PM 并行提交方案
3. 整体规划 PM 归并分歧并输出唯一结论

### 6.2 每轮只允许回答的 4 个问题

- 用户现在要完成什么
- 当前页面真正的问题是什么
- 推荐怎么改
- 明确放弃什么

超出这 4 个问题的延伸讨论，一律视为发散。

### 6.3 每轮产出要求

每轮结束时必须同时满足：

- 形成 1 个推荐方案
- 明确至少 1 个不做项
- 明确下一轮唯一问题
- 给出验收标准
- 明确本轮结论是否已可执行

### 6.4 停机条件

满足以下条件时立即停止迭代，进入文档化阶段：

- 三方对学习主线一致
- 三方对页面职责边界一致
- 三方对第一批改造页面一致
- 每个关键改动都有验收标准
- 已明确至少 3 个不做项

### 6.5 强制停止条件

若到第 3 轮仍存在分歧：

- 由整体规划 PM 直接裁决
- 冻结方案
- 不再开启新方向

## 7. 统一成功标准

所有角色共用以下成功标准：

- 用户能在首页立即知道先看什么
- 页面主职责单一
- 用户可以从首页连续走到主线章节
- 信息负担明显下降
- 证据入口后置，不抢引导入口
- `/analysis` 与 `/sources` 的证据中心定位明确

以下内容不得作为主要成功标准：

- 页面更“好看”
- 信息更“丰富”
- 视觉更“有设计感”

## 8. 最终主文档结构

最终只输出一份主文档，结构固定如下：

1. 目标与问题定义
2. 用户学习主线
3. 页面职责划分
4. UI 交互原则
5. 内容引导原则
6. 第一批改造项
7. 验收标准
8. 附录：轮次记录、分歧与裁决、不做项

## 9. 评审和测试场景

每次输出统一按以下场景自检：

- 新用户第一次进入首页，是否知道唯一推荐入口
- 用户从 `/` 到 `/map` 再到 `/learn/[slug]`，是否能连续走主线
- 用户进入 `/analysis` 或 `/sources` 时，是否知道这是证据中心，而不是第一学习入口
- 单个章节页是否只承担一个主职责
- 首屏是否仍有可删除但未删除的信息块
- 移动端是否能在不展开大量折叠区的情况下完成第一轮学习

## 10. 默认不做项

除非后续明确改动，本轮默认不做：

- 增长和转化策略
- SEO 讨论
- 社区传播和分享机制
- 真实代码实现细节
- 新增后端能力
- 视觉品牌系统重建
