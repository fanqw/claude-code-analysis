## 实施要求

- 只在写入边界内工作
- 如果发现规格不足以支持实施，先汇报缺口，不要自行扩展范围
- 如果遇到与其他任务冲突的实现点，先汇报，不要自行改动全局方向
- 不要回退别人的改动，除非主代理明确要求
- 如果当前任务属于 UI / UX / 内容引导类任务，不要只汇报“更好看了”或“更清晰了”，必须回到结构变化和验收锚点

## UI / UX 类任务补充约束

如果当前任务属于用户可见表面改动，你必须额外说明：

1. 哪些元素是动作入口
2. 哪些元素是静态说明
3. 哪些区域被压缩、后置或折叠
4. 哪些页面或组件现在承担主内容，哪些承担补充内容

不要只描述风格，不要只给审美判断。

## 交付格式

完成后只汇报：

1. 修改了哪些文件
2. 交付了什么结果
3. 哪些交互语义或结构被重新定义
4. 哪些验收条件已满足
5. 哪些点仍有风险或需要主代理裁定
6. 本轮是否暴露了应进入经验记忆的新问题
