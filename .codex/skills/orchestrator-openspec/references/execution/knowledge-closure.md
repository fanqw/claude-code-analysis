# 知识闭环

这份文档回答两个问题：

- AI 犯错之后，主代理应该怎样把错误转化为长期资产
- 哪些经验应该留在项目层，哪些经验应该升级为通用资产

它不替代验收规则。  
它只负责把“这次犯错”变成“下次少犯错”。

## 一、闭环目标

知识闭环的目标不是把所有过程都写下来，而是把可复用的教训和修正稳定回灌。

主代理应追求的结果不是：

- 这轮终于做完了

而是：

- 这轮暴露的问题，下次更不容易重犯

## 二、触发条件

满足任一情况时，主代理应考虑启动知识闭环：

- 子代理结果被判定为 `revise`
- 验证失败暴露了新的错误模式
- 同类错误重复出现
- 当前任务临时补了一条高价值规则
- 当前任务新发现了项目关键约束
- 某个派单模板或验证策略明显失效

不是只有严重事故才需要沉淀。  
高频小错同样值得进入闭环。

## 三、最小闭环步骤

主代理默认按这四步做闭环：

1. 记录错误
2. 解释原因
3. 提炼修正
4. 选择落点

分别回答：

1. 错在哪里
2. 为什么会错
3. 下次应怎样规避
4. 这条经验应写到哪里

如果只记录“错了”，没有解释原因和修正方式，就不算闭环。

## 四、经验记录模板

每条经验至少包含：

- 现象：这次具体错了什么
- 成因：缺了什么信息、约束或验证
- 修正：下次应该加什么检查或提示
- 边界：这条经验适用于哪些任务
- 落点：项目知识、通用 reference，还是仅保留在本轮

推荐优先写短而清楚的经验条目，不要追求长文。

## 五、落点选择

### 1. 留在本轮

适用情况：

- 一次性问题
- 不会复用
- 已被规格或任务包完全吸收

### 2. 留在项目层经验

适用情况：

- 某个项目或系统的特殊约束
- 某条业务链路的常见坑
- 某个仓库特有的验证顺序

### 3. 升级为通用资产

适用情况：

- 跨项目都高频出现
- 会持续影响派单、验收或阶段判断
- 已经证明不是偶发现象

如果升级为通用资产，优先进入 reference、模板或稳定流程说明。

## 六、闭环和验收的关系

验收回答的是：

- 这轮能不能收

知识闭环回答的是：

- 以后怎样更稳地收

即使当前轮最终 `accepted`，如果中途暴露了高价值经验，也仍然可以做知识闭环。

因此：

- `accepted` 不代表不需要沉淀
- `revise` 也不代表一定要升级为长期规则

主代理必须做判断，而不是机械归档。

## 七、闭环和记忆模型的关系

知识闭环主要处理经验记忆。

但当某条经验已经足够稳定时，可以发生两种升级：

1. 升级为项目级长期知识
2. 升级为通用 reference 或规范约束

升级动作必须谨慎，但不能缺失。  
否则体系永远只会“当场补救”，不会形成复利。

## 八、主代理自检问题

轮次结束前，主代理至少自问：

- 这轮有没有暴露新的错误模式
- 这类错误是否已经重复出现
- 下次如果不补知识，是否大概率还会再犯
- 这条经验属于项目特例还是通用模式

如果四个问题里至少两个回答为“是”，通常就该做知识闭环。
