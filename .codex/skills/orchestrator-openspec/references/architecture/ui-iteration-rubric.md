# UI / UX 迭代准则

这份材料只服务一类任务：

- 用户可见表面需要明显调整
- 目标里包含“更清晰”“更易理解”“更统一”“更轻”“更有引导性”这类高主观性描述

它的作用不是指导具体审美，而是把高主观性任务压缩成可派单、可验收的结构。

## 1. 什么时候需要用这份准则

满足任一条件就应先看：

- 涉及入口页、导航、详情页、表单、列表、阅读页等用户可见表面
- 目标里既有视觉诉求，也有信息引导诉求
- 需要区分“动作元素”和“静态说明”
- 需要减少信息轰炸、提升主次层级
- 需要引入新的视觉语言，但不能破坏现有证据层或核心行为

## 2. 主代理在冻结方案前必须锁定的内容

至少锁定以下 6 项：

1. `目标表面`
   哪些页面、组件或区域属于本轮修改范围。

2. `不改动边界`
   哪些能力不能碰，例如真实证据层、路由结构、数据模型、关键交互。

3. `交互语义规则`
   哪些元素是可点击动作，哪些元素是静态说明，哪些元素是页内锚点。

4. `主次层级规则`
   首屏必须先强调什么，次级内容必须放到哪里，哪些内容要折叠或后置。

5. `内容密度策略`
   哪些文案必须压短，哪些区域允许展开，哪些区域只保留一句话。

6. `验收锚点`
   主代理如何判断这一轮确实比之前更清晰，而不是只换了皮肤。

## 3. 推荐的任务拆分方式

UI 类任务不要直接按“页面 A / 页面 B / 页面 C”拆。

优先按下面 4 类拆：

### A. 结构审查

适合先派 `explorer`

关注：

- 当前最影响理解的 3 到 5 个问题
- 这些问题落在哪些文件或区域
- 如果不改，会卡住哪条用户路径

### B. 语义冻结

由主代理负责

要冻结：

- 动作元素样式规则
- 静态标签样式规则
- 页内锚点样式规则
- 主内容区与补充内容区的边界

### C. 样式实施

适合派 `builder`

关注：

- 全局样式变量
- 关键表面组件
- 局部页面文案压缩

### D. 结构验证

适合派 `verifier`

关注：

- 构建是否通过
- 页面是否可访问
- 关键文案、类名、路由是否符合预期

## 4. UI 类任务的验收锚点

至少从这 4 类里选 2 类：

### 1. 点击语义

- 动作元素和静态元素一眼能区分
- 不可点击元素不会伪装成按钮
- 页内锚点不会和主导航竞争

### 2. 主次关系

- 用户进入页面 5 秒内知道先看什么
- 首屏只突出 1 个主动作或 1 条主路径
- 补充内容不会抢首屏注意力

### 3. 文案密度

- 首屏没有连续大段说明文字
- 标题、短说明、补充说明层级清楚
- 折叠区和展开区分工明确

### 4. 风格一致性

- 关键表面使用同一套视觉语义
- 新风格不会只落在 1 个局部，导致全站割裂
- 风格增强不会破坏可读性和可用性

## 5. 子代理输出要求

### Explorer

不能只说：

- “风格不统一”
- “层级不清楚”
- “看起来太乱”

必须输出：

1. 最关键的 3 到 5 个问题
2. 每个问题对应的文件或区域
3. 每个问题影响哪条用户路径
4. 对应的最小修改方向

### Builder

不能只说：

- “我已经做了一版更好看的样式”

必须输出：

1. 修改了哪些文件
2. 哪些交互语义被重新定义
3. 哪些文案被压缩或后置
4. 哪些验收锚点已满足

## 6. 这份准则不负责什么

- 不替代主代理做审美裁定
- 不规定具体视觉风格
- 不决定用哪种色彩或字体
- 不要求 UI 任务必须引入截图

它只负责把“主观目标”压缩成“可执行、可验收的结构”。
