# 记忆模型

这份文档回答两个问题：

- 这套 skill 到底在维护哪些“记忆”
- 不同记忆分别应该落在哪些工件上

它不替代阶段门禁，也不替代 OpenSpec。  
它只负责把 Agentic Engineering 中容易混在一起的记忆分层。

## 一、为什么要分记忆层

长任务失败，很多时候不是因为模型不会做，而是因为记忆被混用了：

- 把稳定约束当成临时说明
- 把本轮过程当成长期知识
- 把一次性踩坑当成已经制度化的规则

主代理必须区分三类记忆：

1. 规范记忆
2. 执行记忆
3. 经验记忆

如果不分层，长任务会出现三类典型问题：

- 规格漂移
- 轮次失焦
- 同类错误反复发生

## 二、规范记忆

规范记忆指当前任务中相对稳定、应成为执行前提的内容。

典型内容：

- 当前 change
- 当前 spec delta
- 当前 design
- 当前 tasks
- 已冻结的目标、范围、约束
- 已确认的行为契约和验收口径

默认载体：

- OpenSpec 工件
- 明确冻结的任务包

规范记忆的使用原则：

- 规范记忆优先于口头说明
- 中大型任务优先以 OpenSpec 工件为准
- 规范一旦变化，必须显式回到阶段门禁重新裁定

规范记忆不能偷偷漂移。

## 三、执行记忆

执行记忆指为了让当前轮和下一轮顺利衔接而保留的运行态信息。

典型内容：

- 当前轮唯一目标
- 当前阶段
- 当前任务包来源
- 当前已完成情况
- 当前阻塞项与风险
- 当前验证状态
- 下一步推荐动作

默认载体：

- 轮次总结
- 上下文包
- 主代理收口记录

执行记忆的使用原则：

- 只保留继续推进所需的最小必要信息
- 轮次切换时必须更新
- 过期执行记忆必须压缩或丢弃

执行记忆不是长期知识库。

## 四、经验记忆

经验记忆指本轮新暴露出来、未来还会重复出现的错误模式和修正经验。

典型内容：

- Agent 常犯错误
- 本轮新增的验证关注点
- 某类任务的高频遗漏项
- 某个系统或项目的关键约束
- 某种派单方式为什么失效
- 某类验证为什么必须前置

默认载体：

- 项目知识文档
- 经验沉淀文档
- 后续会被反复引用的 reference

经验记忆的使用原则：

- 只有可复用经验才沉淀
- 经验必须带来源和适用边界
- 经验沉淀后，下一轮应能被主代理主动引用

经验记忆不是吐槽记录，也不是原始聊天日志。

## 五、三类记忆的关系

推荐按这个方向理解：

- 规范记忆回答“应该按什么做”
- 执行记忆回答“当前做到哪了”
- 经验记忆回答“以后怎样少犯错”

它们之间的流动关系通常是：

1. 先由规范记忆约束执行
2. 再由执行记忆承接当前轮推进
3. 再从执行中提取经验记忆
4. 必要时把成熟经验升级为规范或稳定 reference

不是所有经验都要升级为规范。  
只有已经足够稳定、会持续影响判断的经验，才应进入长期约束层。

## 六、主代理的最小职责

每轮结束时，主代理至少应能明确指出：

- 当前规范记忆入口是什么
- 当前执行记忆包在哪里
- 本轮是否新增经验记忆
- 新经验应该留在项目层，还是升级为通用 reference

如果这四个问题答不清，说明记忆治理还不稳定。

## 七、与 OpenSpec 的关系

OpenSpec 主要承载规范记忆。

这套 skill 不能把 OpenSpec 当成全部记忆系统，因为：

- OpenSpec 不负责记录每轮运行态细节
- OpenSpec 也不负责吸收所有经验性踩坑

因此主代理必须同时维护：

- OpenSpec 规范工件
- 轮次执行记忆
- 经验回灌链路

这样才能把“规格先于执行”和“知识复利”同时成立。
