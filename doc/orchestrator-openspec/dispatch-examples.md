# 最小派单案例集

这份文档只保留最有代表性的 4 类案例。

目的不是重复 `dispatch-playbooks.md` 的规则，而是帮助主代理快速找到“和当前问题最像的起手式”。

如果需要完整规则，先看 [../core/dispatch-playbooks.md](../core/dispatch-playbooks.md)。

## 1. 轻量局部实现

问题特征：

- 目标清楚
- 写入范围清楚
- 风险较低
- 单轮可收口

起手式：

- 直接派 `builder`
- 若出现运行面风险，再补 `verifier`

主代理重点：

- 明确写入范围
- 防止越界修改

## 2. 先探后建

问题特征：

- 目标明确，但实现路径不明确
- 需要先摸清边界

起手式：

1. 先派 `explorer`
2. 主代理冻结方案
3. 再派 `builder`

主代理重点：

- explorer 结果未审完前，不启动 builder

## 3. 实现加技术验收

问题特征：

- 会影响构建、启动、命令、接口或状态流
- 不能只靠静态 diff 判断

起手式：

1. `builder` 实现
2. 主代理初审
3. `verifier` 独立验证
4. 主代理裁定

主代理重点：

- 不用 builder 自测替代 verifier

## 4. 中大型重构

问题特征：

- 跨多个文件或多个子系统
- 需要规格冻结
- 预计至少一轮独立验证

起手式：

1. 主代理先推进 OpenSpec
2. `explorer` 澄清边界
3. 主代理冻结方案和任务包
4. `builder` 实现
5. `verifier` 验证
6. 主代理综合裁定

主代理重点：

- 不从口头说明直接派单
- 必须从 change / tasks 工件派生任务

## 5. 冲突结果

问题特征：

- builder 与 verifier 结论不一致
- 或主代理仍无法确认关键路径是否可信

起手式：

- 不继续默认加派普通角色
- 主代理优先亲自复核

关键原则：

- 冲突处理不外包
