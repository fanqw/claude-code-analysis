# 能力模型

这份文档回答一个问题：

- 这套 skill 的系统级能力到底是什么

它不重复具体派单、验收或 OpenSpec 操作细节，只定义这套主代理编排方法的六项核心能力。

这六项能力同时构成一套 Agentic Engineering 的元工程基础设施：

- 用规格约束 Agent
- 用执行记忆维持长任务稳定
- 用经验回灌形成知识复利
- 用门禁和验证把协作变成工程系统

## 1. 统一执行内核

这套 skill 的第一项能力，是把复杂交付收束到一个统一执行内核里。

这里的“统一执行内核”指：

- 所有工作都围绕主代理推进
- 所有阶段都挂在同一个六阶段状态机上
- 所有派单都回到同一个 change、任务包和验收链路
- 不允许不同子代理各自形成独立流程

落地方式：

- 由主代理锁定每轮唯一目标
- 由主代理维护阶段切换
- 由主代理统一派单、验收、归档

这保证系统不是“多人并行的消息堆”，而是“单内核驱动的多角色协作”。

## 2. 规格先于执行

第二项能力，是让规格成为执行前提，而不是执行后的解释。

这里的关键不是“文档很多”，而是：

- 未冻结目标，不进入实现
- 未冻结方案，不进入并行执行
- 未冻结任务包，不启动 builder
- 中大型任务先进入 OpenSpec change 流程

落地方式：

- 轻量任务可以走轻流程
- 中大型任务必须先过 `proposal / design / tasks`
- 主代理只能从规格工件派生派单和验收

这项能力保证执行不是即兴发挥，而是受规格约束。

## 3. 权限与裁定门禁

第三项能力，是把“谁可以做决定”变成正式系统规则。

这里的权限，不只指环境权限或命令权限，也包括裁定权限。

核心边界：

- 子代理只能完成被分配的局部任务
- 子代理不能决定全局方向
- 子代理不能宣布整体完成
- 只有主代理可以给出最终 `accepted / revise / blocked`
- 遇到关键冲突或高风险结果时，必须回到主代理裁定

落地方式：

- 固定角色边界
- 固定阶段门禁
- 固定主代理最终裁定权

这项能力保证协作不会变成“谁先说完成就算完成”。

## 4. 长任务上下文治理

第四项能力，是把复杂交付当成长任务来治理，而不是只看当前一轮输入。

这里的上下文治理包括：

- 当前轮唯一目标
- 当前阶段位置
- 当前 change 状态
- 当前任务包来源
- 当前证据链
- 当前阻塞项
- 下一步推荐动作

落地方式：

- 每轮都要求最小证据链
- 每轮都要求轮次总结
- 每轮都要求清楚的下一步
- 阶段切换必须有进入条件和退出条件

这项能力的目的，是避免长任务在多轮协作中失去方向。

## 5. 协议化扩展与角色接入

第五项能力，是让扩展通过协议接入，而不是靠临场约定。

这里的“协议化”体现在：

- 角色协议
- 派单协议
- 验收协议
- 生命周期协议
- 模型选择协议
- OpenSpec 工件接入协议

落地方式：

- 标准角色固定为 `builder / explorer / verifier`
- 标准派单字段固定
- 标准验收裁定固定
- 标准生命周期与公告规则固定
- 标准 prompt、脚本和 routing 入口固定

这样以后新增角色、模板或脚本时，都是在协议内扩展，而不是重新发明一套协作方式。

## 6. 证据驱动验收与可追溯性

第六项能力，是让系统对“为什么这样裁定”始终有证据支撑。

核心原则：

- 验收必须回到任务包和规格
- 裁定必须回到验证结果和证据
- 结论必须可追溯到 change、任务、验证和轮次总结

落地方式：

- 先用低成本、可复查证据
- 关键结果必须独立验证
- 每轮至少保留最小证据链
- OpenSpec 工件、验收结果和轮次总结之间要能互相回链

这项能力的目标，不是让文档变多，而是让后续任何人都能回答：

- 为什么这么做
- 为什么这样验收
- 为什么现在可以收口

## 六项能力之间的关系

推荐按这个顺序理解：

1. 先有统一执行内核
2. 再有规格先于执行
3. 再建立权限与裁定门禁
4. 然后治理长任务上下文
5. 再通过协议化方式扩展角色与能力
6. 最后用证据驱动验收和追溯闭环

如果缺少其中任一项，这套 skill 都会退化：

- 没有统一执行内核，会变成多线程散协作
- 没有规格先于执行，会变成即兴实现
- 没有裁定门禁，会变成责任漂移
- 没有上下文治理，会变成长任务失焦
- 没有协议化扩展，会变成每轮重新协商
- 没有证据驱动验收，会变成口头结论系统
