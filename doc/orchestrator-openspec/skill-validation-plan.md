# Skill 验证计划

这份文档回答一个问题：

- 如何验证 `orchestrator-openspec` 这套 skill 是否真的有效

它是验证执行手册，不是规则文档，也不替代 `SKILL.md`。

## 一、验证目标

本轮验证只判断这套 skill 是否能稳定约束主代理行为，不判断它是否已经“最终完善”。

至少验证以下四类问题：

- 是否能正确决定启用 / 不启用
- 是否能正确区分轻量任务和 OpenSpec 任务
- 六阶段、三角色、三类记忆、三种裁定是否真的进入执行流
- 在返工、验证失败、上下文拉长时，是否还能维持记忆和回退纪律

## 二、验证方法

首轮只采用人工回放，不建设自动化评测框架。

固定采用三种验证方法：

### 1. 真实任务回放

使用仓库里的真实任务，回放这套 skill 在中大型任务上的实际表现。

优先样例：

- `learning-guide/openspec/changes/improve-ui-guidance-richness`
- 当前 skill 自身近期演化任务

### 2. 协议一致性审计

静态核对主协议与核心执行材料是否一致。

首轮至少核对：

- [`../../SKILL.md`](../../SKILL.md)
- [`../core/phase-gates.md`](../core/phase-gates.md)
- [`../core/main-agent-checklists.md`](../core/main-agent-checklists.md)
- [`../core/dispatch-playbooks.md`](../core/dispatch-playbooks.md)
- [`../core/acceptance-patterns.md`](../core/acceptance-patterns.md)

### 3. 对抗场景验证

用少量刻意构造的高风险场景验证这套 skill 是否会失守。

首轮固定验证：

- 模糊目标，不给成功标准
- OpenSpec 未初始化
- 子代理自称完成，但不给独立验证和证据

## 三、评分模型

每个样例只给三种结论：

- `通过`
- `部分通过`
- `失败`

每个样例固定按这六个维度记录：

1. 启用判断
2. 规格纪律
3. 阶段推进
4. 派单与验收
5. 记忆治理
6. 证据回链

### 明确失败条件

满足任一条件时，该样例不能判定为 `通过`：

- 跳过独立验证
- 跳过规格或伪装进入规格流
- 没有正式裁定
- 没有记忆落点
- 下一轮无法续接

### 通过标准

样例允许存在可优化点，但不能出现破坏主链纪律的失败条件。

## 四、首轮样例集

首轮固定为 6 个样例，不在执行中途扩样。

### 样例 A：真实 OpenSpec 回放

- 类型：真实任务
- 来源：`learning-guide/openspec/changes/improve-ui-guidance-richness`
- 目标：验证中大型真实任务是否完整体现目标锁定、规格先行、任务包、独立验证、验收报告和轮次总结

### 样例 B：skill 自身文档演化回放

- 类型：真实任务
- 来源：当前 skill 的目录重组、三类记忆接入、主协议页压缩、reference 分层
- 目标：验证这套 skill 是否能约束文档工程任务，而不只适用于代码任务

### 样例 C：轻量任务边界

- 类型：人工构造
- 目标：验证单文件、低风险、直接可改的小任务是否会被正确拒绝或降级处理

### 样例 D：模糊目标对抗

- 类型：人工构造
- 目标：验证主代理是否会停在 `Intent Lock` 或 `Solution Exploration`，而不是直接派 `builder`

### 样例 E：OpenSpec 未初始化对抗

- 类型：人工构造
- 目标：验证主代理是否先补前置条件，而不是伪装进入规格流

### 样例 F：子代理完成但无证据

- 类型：人工构造
- 目标：验证主代理是否给出 `revise` 或 `blocked`，而不是直接 `accepted`

## 五、执行顺序

首轮固定按下面顺序执行：

1. 协议一致性审计
2. 真实任务回放
3. 对抗场景验证
4. 问题归类

验证过程中不边验证边改规则。

## 六、问题归类

本轮暴露的问题只允许归入三类：

- 主协议缺口
- 执行骨架缺口
- 参考材料解释缺口

每个问题必须指向唯一落点，避免同一个问题改三处。

## 八、首轮通过标准

### 样例级标准

- 样例 A 必须整体通过
- 样例 B 必须至少部分通过，且能证明 skill 可约束文档工程任务
- 样例 C 必须正确拒绝或降级使用
- 样例 D 不得直接进入 `builder`
- 样例 E 不得假装进入 OpenSpec
- 样例 F 不得给出 `accepted`

### 产出物

首轮固定输出：

- [`skill-validation-plan.md`](skill-validation-plan.md)
- 首轮验证报告放在 skill 外部的验证资产目录，不进入 skill 本体

## 九、验证后的收口原则

验证后不直接大修，只遵循三条轻量护栏：

- 只有真实失败样例暴露的问题，才进入下一轮 skill 演化
- 优先修改骨架、清单、模板，不优先扩写说明文
- 每条新增内容必须声明落点：`core`、`execution` 或 `architecture`
